# Cendorq Command Design Operating Standard

Cendorq should feel like an AI-readiness command system, not a normal agency website.

The full product direction is:

- Apple-level trust and authority
- Google-level simplicity
- ChatGPT-level immediate action

This standard applies to public pages, protected customer surfaces, internal operator routes, report shells, copy, navigation, validation scripts, SEO/share surfaces, and future product expansion.

## Core rule

Every important surface must help the user answer one question quickly:

> What is the safest next readiness move?

The answer should usually be one of these:

1. Start Free Scan.
2. Compare Plans.
3. Open Dashboard.
4. Open Intake Console.
5. Ask support only when fit, scope, access, or timing needs a human path.

## Apple-level trust and authority

Cendorq earns trust through restraint, clarity, and confidence.

Required behavior:

- Use strong hierarchy.
- Use fewer, better words.
- Keep visual language premium, serious, and intentional.
- Make the offer feel protected, not desperate.
- Avoid fake certainty, fake guarantees, and overpromising.
- State boundaries clearly: first signal, evidence-backed review, scoped repair, monthly control.

Forbidden behavior:

- No vague audit theater.
- No loud agency language.
- No ranking promises.
- No cluttered choice sets.
- No retired public labels.
- No public exposure of private scoring, evidence, report, or operator logic.

## Google-level simplicity

Cendorq should be easy to operate before it is impressive.

Required behavior:

- Keep primary navigation short.
- Use current route language: Free Scan, Plans, Dashboard, Support.
- Use one dominant action per surface.
- Let secondary actions support the main path, not compete with it.
- Prefer direct route labels over clever labels.
- Keep the buyer path visible: Scan, Review, Repair, Control.

Forbidden behavior:

- Do not promote legacy pricing routes.
- Do not split attention with redundant CTAs.
- Do not bury Free Scan behind explanation.
- Do not use labels that require internal context to understand.

## ChatGPT-level immediate action

Cendorq should let a serious visitor begin immediately.

Required behavior:

- Free Scan must remain the clearest first action.
- Forms should appear early when the page purpose is intake.
- Dashboard routes should make the next safe action obvious.
- Protected surfaces should explain state without exposing private logic.
- Internal operator surfaces should prioritize signal, risk, routing, and integrity.

Forbidden behavior:

- Do not make users read the whole site before action.
- Do not make the report shell pretend to be a full engine before it is ready.
- Do not make generic contact the first move when the problem is unclear.

## Public path

The public buyer path is:

1. Home
2. Free Scan
3. Plans
4. AI Readiness Review
5. Signal Repair
6. Readiness Control

The public path sells the outcome and routes the user. It must not expose protected scoring internals, private report evidence, private prompts, operator logic, customer-owned data, or command-center internals.

## Protected path

The protected path owns the deeper system:

- Dashboard
- Reports
- Free Scan result
- Billing
- Notifications
- Support
- Intake Console
- Report shells and future report rendering
- Operator and command-center surfaces

Protected surfaces should be stable, guarded, current-language aligned, and clear about what is available now versus what expands later.

## Copy standard

Use this language:

- AI Engine Readiness
- Free Scan
- Plans
- AI Readiness Review
- Signal Repair
- Readiness Control
- Readiness Path
- Scan, Review, Repair, Control
- easier to find, understand, trust, and choose
- first signal
- safest next action

Avoid this language on active public surfaces:

- Pricing as the primary nav label
- Search Presence Scan
- Visibility Blueprint
- Presence Infrastructure
- Presence Command
- Start Search Presence Scan
- Deep Review
- Build Fix
- Ongoing Control
- Diagnose
- Fix as a plan label
- vague audit theater
- ranking guarantees

## Design standard

Cendorq should feel precise, premium, and controlled. Use depth, focus, spacing, and hierarchy to create authority. Avoid decorative overload.

Every major page should pass this test:

1. Is the main action obvious within seconds?
2. Is the route label current?
3. Is the page easy to scan?
4. Does the copy respect public/private boundaries?
5. Does it make Cendorq feel more trusted, not louder?

## Engineering standard

Every outside change should have an inside guard where practical.

Examples:

- Public label changes should be guarded by validation.
- Route changes should be guarded by route validation.
- Private/report shells should use `noIndex` unless intentionally public.
- Legacy language should be forbidden in active public surfaces.
- New readiness surfaces should document the safest next action and recovery path.

## Operating principle

Do not optimize Cendorq for decoration. Optimize it for readiness clarity.

A user should feel:

> This knows what matters, shows me the safest next move, and does not waste my time.
