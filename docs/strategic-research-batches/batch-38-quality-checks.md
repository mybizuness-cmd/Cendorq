# Cendorq Strategic Research Batch 38: Quality Checks

Cendorq needs repeatable quality checks for customer-facing reports, dashboards, PDFs, emails, and paid-work artifacts.

## Check targets

- report package structure;
- evidence visibility level;
- report lineage;
- customer-safe wording;
- plan routing;
- dashboard, PDF, and email consistency;
- visual module usefulness;
- entitlement state;
- delivery event completeness;
- unsupported outcome language;
- private internal data boundaries.

## Required checks

- each customer-facing finding has evidence context or clear uncertainty;
- confidence is included where needed;
- dashboard and PDF tell the same story;
- plan CTA follows the evidence;
- competitor statements are supported;
- Build Fix completion has a completion artifact;
- Ongoing Control snapshot includes period and drift summary;
- visuals explain a business decision.

## Agent instruction

- Quality checks are part of release control.
- If a check fails, repair the source or split the implementation batch.
- Do not reduce the standard just to finish faster.
- A report that sounds polished but lacks evidence, access clarity, or artifact consistency is not ready.

## Final command

Quality checks turn doctrine into repeatable release control. Cendorq should ship customer-facing artifacts only when the source, language, visual, plan route, and delivery state are aligned.
