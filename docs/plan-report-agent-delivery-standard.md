# Cendorq Plan Report and Agent Delivery Standard

Use this standard for customer reports, paid plan workflows, agent missions, customer intake, dashboard report records, report visuals, forecast notes, delivery summaries, and next-plan recommendations.

The goal is simple: every Cendorq report should be clear, premium, visual, evidence-aware, and easy for a business owner to understand.

## Core principle

Reports must be educational while selling and informational while selling.

Every report must:

- teach the customer what matters
- show the evidence behind the finding
- explain the business meaning in plain words
- use helpful visuals when visuals make the point easier to understand
- separate facts, assumptions, inferences, limitations, and recommendations
- avoid unsupported outcome promises
- recommend the next plan only when the evidence supports it

## Plan-to-report map

### Free Scan

Purpose: create a useful first signal and make the customer understand why a deeper review may be worth it.

Maximum customer-facing length: two pages.

Required customer input:

- business name
- website or main public profile
- business type
- location or service area when relevant
- main offer
- target customer
- main concern or goal

Required report output:

- Cendorq letterhead
- customer business name
- report title: Free Scan Signal Report
- generated date
- short executive signal
- 3 to 5 key findings
- simple visibility/readiness signal level
- plain explanation of what the findings mean
- limitation note
- next action
- upgrade path to Deep Review

Free Scan visual requirements:

- simple signal meter or readiness bar
- small issue-priority visual
- no full competitor table
- no full forecast model
- no full fix roadmap

### Deep Review

Purpose: deliver the full scan, full diagnosis, and cause-level visibility/readiness review.

Required customer input:

- active Deep Review entitlement
- verified customer ownership or authorized requester
- website
- business name
- business type
- target customer
- main offer
- main conversion action
- location or service area when relevant
- social or profile links when relevant
- known competitors when available
- current channels or platforms when relevant
- what the customer has already tried
- biggest concern

Required report output:

- Cendorq letterhead
- customer business name and safe logo status
- report title: Deep Review Diagnostic Report
- generated date and report version
- table of contents
- executive summary
- visibility and readiness scorecard
- website clarity review
- AI/search readability review
- trust and proof review
- offer and customer-action review
- competitor comparison when evidence is available
- forecast or risk outlook
- priority map
- evidence-backed findings
- confidence labels
- limitations
- recommended next action
- Build Fix fit decision
- support and correction path

Deep Review visual requirements:

- visibility/readiness scorecard
- priority heat map
- competitor comparison chart when evidence allows
- forecast/risk outlook visual
- opportunity map when useful
- clear severity labels

### Build Fix

Purpose: turn an approved weakness into a scoped implementation plan, delivery summary, and progress expectation.

Required customer input:

- active Build Fix entitlement
- verified customer ownership or authorized requester
- approved fix target
- approved scope
- primary call to action
- brand guidelines or tone preference when available
- logo or assets when needed
- access handoff method when required
- list of things that must not change
- approval contact

Required report output:

- Cendorq letterhead
- customer business name and safe logo status
- report title: Build Fix Delivery Report
- generated date and report version
- approved scope
- before-state baseline
- what was fixed or prepared
- why it matters
- expected timeline range for movement
- dependencies and limitations
- remaining risks
- out-of-scope items
- recommended next action
- Ongoing Control recommendation when fit
- support and correction path

Build Fix visual requirements:

- before/after summary visual
- scoped work checklist
- expected timeline visual
- remaining-risk map
- next-control path visual

Build Fix must explain that improvement can take time and depends on implementation, market movement, search behavior, AI systems, platforms, competitors, and customer action.

### Ongoing Control

Purpose: keep the customer from becoming stale as search, AI engines, platforms, customers, and competitors keep changing.

Required customer input:

- active Ongoing Control subscription
- verified customer ownership or authorized requester
- monitoring scope
- review cadence
- monthly or cycle priority
- known business changes
- new offers or service updates
- competitors to watch when relevant
- platforms or channels to watch when relevant

Required report output:

- Cendorq letterhead
- customer business name and safe logo status
- report title: Ongoing Control Report
- cycle date and report version
- executive summary
- what improved
- what weakened
- what stayed stable
- AI/search/platform change notes when relevant
- competitor movement notes when evidence allows
- trend confidence labels
- one main priority for the cycle
- recommended next action
- Build Fix escalation decision when fit
- support and correction path

Ongoing Control visual requirements:

- trend line or month-over-month signal visual
- change summary cards
- risk watchlist
- priority meter
- next-action path visual

## Report design standard

Every customer report must look pristine.

Required visual identity:

- Cendorq letterhead
- report title
- customer business name
- customer logo only when provided, safe, approved, and stored through safe projection
- Cendorq mark or neutral Cendorq placeholder when customer logo is missing
- generated date
- report type
- report version
- release status
- clean page numbers for PDF output
- calm professional color system
- clear section hierarchy
- high-contrast readable text
- mobile-readable HTML version
- polished PDF version when applicable

Required report visuals:

- scorecards where scores help the customer understand priority
- bar charts for simple comparison
- line charts for trend over time when data exists
- heat maps for severity or priority
- easy tables for issue or competitor comparisons
- timeline visuals for Build Fix expectations
- progress visuals for Ongoing Control
- callout boxes for key business meaning

Visuals must teach. Do not add decorative graphs that do not make the decision clearer.

## Customer wording standard

Use words a customer understands:

- visibility
- readiness
- trust
- clarity
- proof
- customer action path
- weak point
- strongest next step
- what this means
- what to fix first
- what may take time
- where competitors look stronger
- what changed this month

Every technical point should answer:

- What did we see?
- Why does it matter?
- How confident are we?
- What should happen next?

## Research standard

Research must be deep, evidence-aware, and practical.

Agents should use the best available safe sources, including customer-provided information, the customer website, public business profiles, search visibility signals, AI/search readability signals, social or platform activity when relevant, reviews or trust cues when relevant, competitor public pages when relevant, maps/local signals when relevant, content and offer clarity, page structure, conversion path, and industry context.

Research must separate verified facts, customer-provided context, external evidence, assumptions, inferences, evidence gaps, confidence level, and recommended action.

When evidence is limited, say so clearly and turn it into a useful next step.

## Dashboard report rules

Every report or report-related output must live in the customer dashboard.

Dashboard report records should include report ID, report type, plan type, business name, generated date, release status, report version, short summary, next recommended action, missing customer inputs when blocked, support and correction path, and view or download link when approved.

The dashboard must not expose draft, unapproved, raw, internal, cross-customer, or unsafe report data.

## Agent chain of command

The report system follows this chain:

1. Owner Command sets business direction and top standard.
2. Release Captain owns final validation and customer-facing approval.
3. Chief Report Truth Agent owns evidence, facts, assumptions, confidence, and report truth.
4. Chief Customer Experience Agent owns plain-language clarity, structure, design quality, and customer comprehension.
5. Chief Security and Privacy Agent owns private data, safe projection, no-secret exposure, and report boundary safety.
6. Chief Market Forecast Agent owns competitor movement, market shifts, AI/search changes, and forecast quality.
7. Scoped sub-agents research, compare, draft, visualize, and pressure-test inside their lanes.

No sub-agent or chief agent can release a customer report without release captain approval.

## Agent quality standard

Every agent finding must include finding, source or evidence basis, business meaning, confidence, limitation, recommended next action, and customer-safe claim check.

Agents must reject generic filler, unsourced claims, fake certainty, fake urgency, unsupported competitor claims, unsupported forecasts, unsupported outcomes, customer-facing technical clutter, and visuals that do not clarify the decision.

## Next-plan motion

Each report should move the customer forward naturally:

- Free Scan should lead to Deep Review when deeper cause-level diagnosis is needed.
- Deep Review should lead to Build Fix when an approved weakness can be improved.
- Build Fix should lead to Ongoing Control when the customer needs to keep visibility and readiness from drifting.
- Ongoing Control should keep the customer current and may recommend another Build Fix when a new scoped weakness appears.

Do not push a plan that does not fit the evidence.

## Blocked report behavior

A report must pause and request more customer context when the missing information would make the report misleading.

Use simple customer-facing wording:

- We need one more detail before we can finish this correctly.
- This helps us avoid guessing.
- Please confirm the best customer action you want people to take.
- Please provide the profile, offer, or location you want reviewed.

Do not make the customer feel blamed. The system should feel precise, helpful, and protective.

## Non-goals

Do not use reports, agents, or plan delivery as a reason to add fake guarantees, fake urgency, generic AI content, technical language customers cannot understand, unapproved customer logo use, raw evidence exposure, private prompt exposure, cross-customer comparisons using private data, full competitor analysis inside Free Scan, untested public message boxes, fake support success states, or reports that are all text and no visual explanation when visuals would help.
