# Acquisition-to-Retention Operating System

Cendorq must operate from before the visit through long-term retention as one connected system. The goal is not to chase traffic with unsupported promises. The goal is to become easier to discover, easier to trust, easier to choose, easier to recover, and harder to replace through proof, clarity, safety, and continuous usefulness.

## Core rule

Every acquisition, conversion, delivery, support, and retention surface must answer:

1. What brought the customer here?
2. What do they need to understand immediately?
3. What proof or boundary makes the next step feel safe?
4. What is the one strongest action?
5. What is the recovery path if they are not ready, confused, blocked, or returning later?
6. What data stays public and what stays protected?
7. What changes over time that gives the customer a true reason to come back?

## Discovery before the visit

Cendorq should pursue discovery through useful public content, clean technical SEO, structured data, accessibility, speed, crawlable public routes, trustworthy page titles, descriptive metadata, public education, clear plan language, and category-defining authority.

Cendorq must not promise guaranteed traffic, guaranteed ranking, guaranteed indexing, guaranteed AI answer placement, guaranteed number-one search results, guaranteed leads, or guaranteed revenue.

Public discovery rules:

- public pages should be indexable only when they are meant to be public
- private dashboards, reports, billing, support, admin, command-center, customer documents, and customer-specific files must not be public discovery surfaces
- public content may explain the category and customer value, but not private prompts, scoring weights, private source weighting, raw evidence, raw provider payloads, raw support records, or customer data
- structured data should support accurate understanding, not fake authority
- public copy should make Cendorq easier to understand and choose, not louder or more desperate
- public pages should be useful even before the customer buys

## Landing experience

Landing pages should feel calm, high-authority, and clear within seconds.

Required landing pattern:

- name the real business problem
- explain why it matters now
- state what Cendorq does in customer language
- give one strongest next action
- give one secondary safe path when needed
- show the boundary: what is not promised and what is not included
- avoid generic agency language, generic AI-tool language, and noisy CTA walls

## Public page roles

- Homepage: category authority and first decision path.
- Free Scan: first signal and protected intake.
- Plans: decision-depth comparison.
- Plan detail pages: one plan, one buyer objection, one checkout path, one fallback to comparison.
- Terms, privacy, disclaimer: trust boundaries that protect without weakening desire.
- Public policy and discovery files: search/AI understanding without private exposure.

## Conversion path

Cendorq should convert by reducing uncertainty, not by adding pressure.

Conversion rules:

- Free Scan should create a useful first signal, not pretend to be a full audit
- AI Readiness Review should sell deeper proof when cause and priority matter
- Signal Repair should sell scoped execution only when the weak signal is clear enough
- Readiness Control should sell ongoing value when change, movement risk, baseline tracking, or forecast refresh gives a true reason to return
- every plan should show included/not-included boundaries
- every stronger recommendation should be tied to evidence
- every checkout path should preserve provider-authoritative billing trust

## After checkout

After checkout, Cendorq must not leave the customer guessing.

Post-payment rules:

- confirm the selected plan
- show what unlocked
- show what happens next
- route to dashboard, billing, messages, report vault, support, or intake as appropriate
- mirror important emails into dashboard messages where applicable
- never make email or PDF the only access path for important customer state
- show document state, report state, billing state, support state, and next action using the same truth structure

## Email and lifecycle nurturing

Emails should be useful, recognizable, low-noise, mobile-readable, and connected to dashboard recovery.

Lifecycle rules:

- verification email opens the protected customer path
- report-ready email returns the customer to the report vault
- billing email returns the customer to billing center or provider-authoritative state
- support email returns the customer to support status or support center
- plan-fit email should educate before selling
- retention email should only appear when there is a real reason: movement risk, forecast refresh, competitor movement, proof freshness, unresolved blocker, new feature relevance, or baseline comparison
- every important email should have a mirrored dashboard message when applicable
- email copy must not promise inbox placement, guaranteed delivery, guaranteed ranking, guaranteed revenue, or guaranteed AI placement

## Dashboard conversion and retention

The dashboard is the private money surface. It should not feel like a generic account page.

Dashboard rules:

- proof first
- next best action second
- source-of-truth recovery always available
- support, billing, report vault, and notifications should return to the same command center
- conversion should be stage-targeted and evidence-backed
- monthly or recurring value should be shown only where it is true
- the customer should always understand what changed, what is pending, what is ready, what is blocked, what is recommended, and what is outside scope

## Report value and forecast usefulness

Reports should create trust by separating fact, assumption, inference, forecast, limitation, and next action.

Report rules:

- forecasts are directional decision aids, not guarantees
- reports should show enough methodology to create trust without exposing the private engine
- report vault remains the canonical protected view
- PDFs must mirror released report-vault state and pass document-safety gates
- update/refresh recommendations should be tied to change, proof freshness, competitor movement, platform behavior, or customer-stage risk

## Support and recovery

Support protects revenue when it resolves anxiety.

Support rules:

- acknowledge the human issue
- show current safe state
- choose source of truth first: dashboard, report vault, billing center, support status, or provider state
- give one next action
- avoid duplicate request loops
- collect safe summaries only
- never ask for passwords, card numbers, private keys, provider secrets, raw attack payloads, or unrelated private evidence

## Retention and reactivation

Retention must be earned by true ongoing usefulness.

Retention reasons:

- search/AI behavior changed
- customer expectations changed
- competitor posture changed
- proof or reviews became stale
- a baseline moved
- a blocker remains unresolved
- a platform or policy changed
- a forecast needs refresh
- a report correction or update is needed
- a new feature can genuinely improve the customer path

Reactivation should be calm and specific: what changed, why it matters, what Cendorq can check, and what the safest next action is.

## Security and privacy across the journey

Growth must not expose private data.

Rules:

- public discovery is for public education and public buyer paths only
- protected customer surfaces require verified access and authorization
- private reports, billing, support, dashboard messages, customer documents, command-center surfaces, and admin surfaces stay protected
- robots.txt is not a privacy control
- noindex/no-store and authenticated access should be used where appropriate
- customer data should not appear in public metadata, public pages, public sitemaps, client bundles, analytics payloads, or logs

## Continuous improvement loop

Cendorq must remain current without uncontrolled mutation.

Operating loop:

1. Research market, customer, platform, security, SEO, AI/search, and competitor changes.
2. Identify which pages, messages, reports, or plan-fit logic are affected.
3. Draft small controlled updates.
4. Validate claims, security, privacy, accessibility, SEO, support, billing, report, and dashboard effects.
5. Review through release-captain posture.
6. Release with rollback awareness.
7. Monitor outcomes and support patterns.
8. Feed safe lessons into doctrine, validators, lifecycle, and report logic.

## Blocked patterns

Cendorq must block:

- guaranteed ranking or traffic promises
- guaranteed indexing or AI answer placement promises
- cluttered page systems
- generic SEO blog spam
- private content used for public discovery
- lifecycle emails with no real reason
- dashboard upsells without evidence
- support loops that avoid source of truth
- reports that blur facts and forecasts
- retention pressure without movement risk
- legal or security language that scares customers without guiding them
- public transparency that exposes private mechanics

## Final rule

Every surface must make Cendorq easier to discover, trust, choose, use, recover, and return to without exposing private data or promising what no serious company can honestly promise.
