# Cendorq Benchmark Intelligence Standard

This standard defines how Cendorq should build a private benchmark intelligence layer for evidence-backed reports, optimization recommendations, and monthly control.

## Principle

Cendorq should learn from strong reference businesses, but it must not copy blindly. Benchmarks are comparison tools, not truth by themselves.

The system should use curated, high-quality reference companies to understand what strong clarity, trust, proof, conversion paths, and operating signals look like in each category.

## Benchmark selection

For each served category, the system should maintain a small reviewed benchmark set.

Default target:

- five approved benchmark businesses per category
- reviewed evidence for every benchmark
- category-specific selection standards
- retirement when a benchmark becomes stale, weak, misleading, or no longer representative

Benchmarks should be selected for:

- strong trust presence
- clear offer
- credible proof
- customer-friendly path
- category relevance
- observable operating quality

## Evidence requirements

Each benchmark should be backed by source evidence such as:

- website copy
- service pages
- local profiles
- review patterns
- visual proof
- policy pages
- booking or contact paths
- product or service proof
- authority content

Every source should be stored with enough metadata to verify where the observation came from and when it was reviewed.

## Comparison usage

The benchmark layer should help compare customer submissions against strong category patterns.

It can support:

- clarity checks
- trust checks
- proof checks
- conversion path checks
- local authority checks
- customer-safe summary checks
- optimization opportunity detection
- monthly progress comparison

## AI self-review loop

The AI manager should evaluate its own work before customer-facing output is approved.

It should mark:

- what is supported
- what is unsupported
- what is uncertain
- what needs more evidence
- what changed from the previous version
- what recommendation is strongest
- what recommendation is too weak to send

AI self-review is not final approval. It is a quality layer before human or operator approval.

## Model and agent evolution

The AI manager should track which model family, version, prompt policy, and evaluation policy produced each important output.

When the AI stack improves, Cendorq should be able to:

- compare old and new outputs
- rerun selected tests
- detect quality regression
- promote better methods
- retire weaker methods
- keep a record of what changed

The source of truth remains Cendorq records and evidence. External AI tools should assist the workflow, not become the system of record.

## Test set requirements

The system should support internal test runs for:

- benchmark comparison
- sample intake scoring
- sample report generation
- optimization recommendation review
- monthly cycle recommendation review
- customer-safe language checks
- evidence completeness checks
- regression checks between model versions

Test records must be separated from live customer records.

## Non-negotiables

- No unsupported claims.
- No benchmark without source evidence.
- No AI recommendation without self-review.
- No customer-facing output without approval.
- No stale benchmark treated as authority.
- No private benchmark notes in customer-facing output.
- No model upgrade without regression checks.
- Cendorq remains the source of truth.
